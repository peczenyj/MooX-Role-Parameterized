# Phase 1 — Minimum Perl 5.12 + Release Automation

- **Date:** 2026-05-15
- **Status:** Approved
- **Phase:** 1 of a 5-phase modernization roadmap for `MooX::Role::Parameterized`

## Context

`MooX::Role::Parameterized` is a CPAN distribution currently targeting Perl 5.008
and built with `ExtUtils::MakeMaker`. The maintainer wants to modernize it. The
work is decomposed into five phases, each with its own spec → plan → implement
cycle:

| # | Phase | Status |
|---|---|---|
| 0 | Drive-by `0.5O1` typo bugfix | folded into Phase 1 release `v0.502` |
| 1 | **Bump min Perl to 5.12 + release automation** | this spec |
| 2 | Tooling & release pipeline (build system, single-source `$VERSION`, auto-README, modern CI, broader author tests) | future |
| 3 | Code modernization under the 5.12 floor (`//`, `state`, reconsider `our %INFO` and `goto &`) | future |
| 4 | Documentation & posture (drop "experimental" framing, `SECURITY.md`, branch reconciliation) | future |
| 5 | Bump min Perl to 5.20+ and adopt signatures / postfix deref | future |

This spec covers Phase 1 only.

## Goals

1. Raise the supported Perl floor from 5.008 to 5.012.
2. Add automated release-to-CPAN and GitHub-release creation, triggered by `v*`
   tags.
3. Introduce `xt/` author tests for the existing perlcritic and perltidy checks.
4. Fix the known `$VERSION` typo in `Mop.pm` (`"0.5O1"` — letter O, not zero).

## Non-goals (deferred to later phases)

- Migrating off `ExtUtils::MakeMaker` (Phase 2).
- Single-sourcing `$VERSION` across the three modules (Phase 2).
- Auto-generating `README.md` from POD (Phase 2).
- Modernizing the `macos.yml` / `windows.yml` workflows or adding a Perl-version
  matrix to them (Phase 2).
- Using any 5.12 language feature in code beyond the `use v5.12;` declaration
  itself (Phase 3).
- Dropping the redundant `use strict;` lines now implied by `use v5.12;`
  (Phase 3).

## Decisions

| Decision | Choice | Rationale |
|---|---|---|
| New Perl floor | 5.012 | Set by maintainer; a second bump to 5.20+ is Phase 5. |
| Version bump for the floor release | `0.600` (minor) | Signals a meaningful install-base change; keeps the 0.x "experimental" posture until Phase 4. |
| Typo fix delivery | Separate `v0.502` patch release first | Rehearses the new release pipeline end-to-end before the larger change. |
| CI matrix (`linux.yml`) | `5.12, 5.20, 5.30` + latest stable | Tests floor, the next planned floor (early warning for Phase 5), a middle version, and head. |
| `use v5.12;` scope | All `lib/`, `t/lib/`, `t/*.t`, and `examples/*.pl` files | Maintainer chose maximum consistency. |
| `use strict; use warnings;` | Retained | `use v5.12;` implies `strict` but not `warnings`; not removing the redundant `strict` line until Phase 3. |
| Release gates | tests + tag/version check + perlcritic + perltidy | All four selected by maintainer. |
| GitHub release visibility | Published immediately (not draft) | Maintainer choice. |
| Lint checks in the release gate | Run via new `xt/` author tests | Maintainer chose to build `xt/` now rather than inline commands. |

## Deliverables

### A. `xt/` author tests

Two new files, run via `prove xt/` — never by `make test` or end users.

- `xt/author/perlcritic.t` — uses `Test::Perl::Critic` configured with the
  existing `.perlcriticrc` profile; critiques `lib/` (matching the current
  `perlcritic lib` scope).
- `xt/author/perltidy.t` — uses `Test::PerlTidy::run_tests` configured with the
  existing `.perltidyrc`; checks the source files (`lib/`, `t/`, `xt/`,
  `examples/`), consistent with the current `perltidy` CI glob
  `**/*.p[lm] **/*.t`.

`MANIFEST.SKIP` already excludes `^xt/` and dotfiles (`^\.`), so `xt/` and the
`.perlcriticrc` / `.perltidyrc` configs are kept out of the CPAN tarball
automatically. No `MANIFEST.SKIP` change is required, and the rc files do not
need to ship.

### B. Develop-phase dependencies

Add to `Makefile.PL` under `META_MERGE → prereqs → develop → requires` (already
consumed by the `cpanm --with-develop` invocations the project uses in CI and
`CONTRIBUTING.pod`):

- `Test::Perl::Critic`
- `Test::PerlTidy`
- `Perl::Critic::Policy::Documentation::RequirePodLinksIncludeText`
- `Perl::Critic::Policy::Miscellanea::RequireRcsKeywords`

The two policy distributions are required so `Perl::Critic` recognizes the
`[-Documentation::RequirePodLinksIncludeText]` and
`[-Miscellanea::RequireRcsKeywords]` disable directives in `.perlcriticrc` —
this is why the current `perlcritic.yml` installs them explicitly.

### C. Convert the lint workflows

- `.github/workflows/perlcritic.yml` — install deps with
  `cpanm --with-develop --installdeps .`, then run `prove -l xt/author/perlcritic.t`.
- `.github/workflows/perltidy.yml` — same, then run `prove -l xt/author/perltidy.t`.

Both remain separate workflows so the existing README status badges still
resolve.

### D. New release workflow — `.github/workflows/release.yml`

- **Trigger:** `on: push: tags: ['v*']`
- **Runner:** `ubuntu-latest`, `container: perl:latest`
- **Secrets required (maintainer adds these in repo settings):**
  `PAUSE_USERNAME`, `PAUSE_PASSWORD`. `GITHUB_TOKEN` is provided automatically.
- **Steps, in order:**
  1. Checkout; `git config --global --add safe.directory` (same workaround the
     other workflows use).
  2. Install deps: `cpanm -nq --with-develop --installdeps .` plus
     `CPAN::Uploader`.
  3. **Gate — version/tag check:** verify the tag minus its leading `v` equals
     the `$VERSION` string in all three modules. Aborts on any mismatch. This
     catches both a tag/code mismatch and the three-files-out-of-sync problem.
  4. **Gate — tests:** `prove -lr t`.
  5. **Gate — author tests:** `prove -l xt/` (perlcritic + perltidy).
  6. **Build:** `perl Makefile.PL && make manifest && make dist`, producing
     `MooX-Role-Parameterized-<version>.tar.gz`.
  7. **Upload to CPAN:** the `cpan-upload` CLI from `CPAN::Uploader`, using the
     PAUSE secrets. No third-party action — minimal trust surface.
  8. **GitHub release:** `softprops/action-gh-release`, attaching the tarball,
     published immediately (not a draft).

### E. The floor bump

- `Makefile.PL`: `MIN_PERL_VERSION` `5.008` → `5.012`.
- Add `use v5.12;` to every Perl source file — all three `lib/` modules, all six
  `t/lib/` helpers, all fourteen `t/*.t` files, and all three `examples/*.pl`
  files. Placed immediately after the `package` statement (where present),
  before `use strict;`. Existing `use strict; use warnings;` lines are retained.
- `.github/workflows/linux.yml`: matrix → `5.12, 5.20, 5.30` + latest stable;
  the coverage job stays on the latest. `macos.yml` and `windows.yml` are
  unchanged (no version matrix; they install whatever `brew` / `choco`
  provide).

`Perl::Critic` natively treats a `use v5.12` statement as satisfying the
`TestingAndDebugging::RequireUseStrict` policy, so no `.perlcriticrc` change is
needed.

## Release sequencing

1. **`v0.502`** — Deliverables A, B, C, and D land in the repository.
   `$VERSION` → `"0.502"` in all three modules (correcting the `0.5O1` typo).
   `Changelog` entry. The PAUSE secrets must be configured before the tag is
   pushed. Tagging `v0.502` triggers `release.yml` for the first time — a full
   live rehearsal of the pipeline. The shipped tarball is functionally just
   "0.501 plus the typo fix"; the automation and `xt/` tests are excluded from
   the tarball.

2. **`v0.600`** — Deliverable E. `$VERSION` → `"0.600"` in all three modules.
   `Changelog` entry. Tagging `v0.600` triggers `release.yml` again.

## Verification

- `perl Makefile.PL && make && make test` passes on a Perl 5.12 build and on the
  latest stable.
- `prove -lr t` passes.
- `prove -l xt/` passes (perlcritic and perltidy clean).
- The `release.yml` version/tag gate succeeds for a correctly tagged release and
  fails for a deliberately mismatched tag.
- `v0.502` appears on CPAN and as a published GitHub release with the tarball
  attached.

## Risks

- **`Test::PerlTidy` vs the current `perltidy -b && git diff --exit-code`
  approach:** equivalent given the same `.perltidyrc`, but `Test::PerlTidy`
  output is sensitive to the installed `Perl::Tidy` version. The current CI
  already carries this exposure (`perl:latest`), so it is not a new risk.
- **`v0.502` is the first real run of `release.yml`:** any gate or CPAN-upload
  misconfiguration surfaces here. This is the intent of doing the small release
  first, but it requires the PAUSE secrets to be set before tagging `v0.502`.
- **Floor bump behavioral surface:** `use v5.12;` enables the 5.12 feature
  bundle (`say`, `state`, `unicode_strings`, `switch`). None are used by the
  code. `unicode_strings` is the only behavioral surface, and the codebase does
  no non-ASCII string manipulation. `switch` (`given`/`when`) only warns if
  used.
- Dropping the 5.8 / 5.10 CI rows means accidental reliance on pre-5.12 behavior
  will not be caught — this is the intended consequence of the floor bump.
